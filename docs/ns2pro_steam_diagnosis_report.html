Steam 最终正确识别虚拟 NS2Pro

根因不是按键 report、gyro 数据或 bulk 协议内容错误，而是 Windows 没有把虚拟设备的 MI_01 vendor bulk interface 以应用可稳定发现的 WinUSB device interface 暴露给 Steam/SDL。 补齐 Microsoft OS 1.0 Extended Properties descriptor 中的 DeviceInterfaceGUID 后，Steam 才开始进入 Switch2 专用路径并访问 ep=2 bulk endpoint。

初始目标

参考 switch2_input_viewer.py 和 bluetooth_interface.md，实现 NS2Pro 蓝牙连接，然后由 VIIPER 模拟成 USB 连接的 NS2Pro 手柄。这样绕过 Windows 不支持 NS2Pro 蓝牙 HID 的限制，让 Steam 在手柄蓝牙连接前就识别到一个虚拟 USB NS2Pro。

实际症状

  • 设备名称正确，按钮可用。
  • Steam 的 Begin Test layout 仍是通用手柄。
  • 缺少 NS2Pro 专属项，如 GL、GR、gyro、LED 等。
  • Go example 曾经“看起来成功”，后来确认是先前真实有线 NS2Pro 留下的设备/Steam 状态造成的假象。

诊断时间线

  • 1
    实现 Python BLE bridge，先创建虚拟 NS2Pro，再等待蓝牙手柄输入。
  • 2
    发现 Steam 仍走 generic layout，且 Go example 冷启动也一样。
  • 3
    补齐真机式复合 USB 描述符：IAD、vendor bulk、audio interfaces、WinUSB compatible ID。
  • 4
    MI_01 从 Error 变为 OK，但 Steam 仍不访问 ep=2。
  • 5
    加入 Extended Properties / DeviceInterfaceGUID，清理旧实例后 Steam 正确识别。

关键证据 A

Get-PnpDevice 显示 MI_01 后来已经是 Status OK，说明 vendor interface 的 Windows 驱动绑定已不再是 Error。

关键证据 B

诊断 trace 起初没有任何 ep=2 dir=OUT，说明 Steam 没有进入 SDL Switch2 USB 初始化路径。

关键证据 C

真实 NS2Pro 有线接入一次后，虚拟设备能被正确识别，说明 Steam/Windows 需要某种真实设备枚举时产生的接口发现状态或缓存。

代码修改摘要

区域 修改 目的
examples/python/virtual_ns2pro_ble_bridge.py 新增单文件 uv 脚本，启动即创建虚拟 NS2Pro，并桥接 BLE input。 让 Steam 在真实蓝牙手柄连接前先看到 USB NS2Pro。
usb/usbdesc.go 增加 configuration、IAD、alternate setting、endpoint class descriptor、MS OS 1.0 支持。 表达真机 NS2Pro 的复合 USB 拓扑。
internal/server/usb/server.go 配置描述符构建支持 IAD、audio alternate interfaces、string 0xEE。 让 Windows 枚举出 HID、vendor、audio 多接口布局。
device/ns2pro/device.go 补齐真机接口布局、WinUSB compatible ID、Extended Properties、audio class control stub。 让 Windows 和 Steam/SDL 能发现并初始化 vendor bulk interface。
device/ns2pro/device.go bulk response 不再按 64 字节拆队列。 避免 SDL 一次 bulk read 读取 0x50 flash 响应时发生残包错位。
device/ns2pro/ns2pro_test.go 新增 SDL USB 初始化序列、MS OS descriptor、配置描述符相关测试。 锁住 Steam 识别依赖的枚举和初始化行为。

Steam/SDL 期望路径

识别 VID_057E PID_2069
获取 libusb/WinUSB device handle
查找 interface 1 的 bulk endpoints
claim interface 1
读取 flash calibration blocks
发送 0x0c / 0x03 初始化命令
切换 report 0x05，启用 gyro/GL/GR/LED

诊断阶段 trace 关注点

临时启用 NS2Pro endpoint trace
ep=2 dir=OUT out=02 91 ...
ep=2 dir=OUT out=0c 91 ...
ep=2 dir=OUT out=03 91 ...

诊断阶段看到 ep=2 记录，就说明 Steam 已经不再只走 generic HID，而是开始执行 Switch2 专用 USB 初始化。最终实现已移除运行时 trace 代码。

注意

Windows 会缓存 USB 设备实例和 Microsoft OS descriptor 结果。因此修改 WinUSB / OS descriptor 后， 需要移除旧的 VID_057E&PID_2069 PnP 实例并重新枚举设备，必要时重启 Windows 和 Steam。

结论沉淀

不要只看名称和 VID/PID

Steam 的高级 layout 依赖 Switch2 专用驱动路径，不是 HID 名称匹配。

WinUSB 可发现性是关键

Composite 设备的 vendor interface 需要 compatible ID 和 device interface GUID。

trace 比猜协议快

没有 ep=2 就不要继续调 bulk payload，先修枚举链路。
